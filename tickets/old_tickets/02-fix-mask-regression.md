# Тикет: Исправить регрессию маски стен после vectorization-pipeline

## Проблема

После интеграции vectorization-pipeline маска стен стала непригодной: почти полностью чёрная с мелкими белыми фрагментами. Стены не распознаются. Раньше (до vectorization-pipeline) маска была нормальной — чёткие белые стены на чёрном фоне.

Скриншот: `image_2026-03-14_12-28-29.png` — видно что маска содержит только шум, стены отсутствуют.

## Корневые причины

### 1. CRITICAL: normalize_brightness() + color_filter() портят изображение перед бинаризацией
- `normalize_brightness()` применяет CLAHE к L-каналу LAB — это усиливает контраст в тёмных областях и меняет распределение яркости
- `color_filter()` с `saturation_threshold=50` слишком агрессивно — на планах эвакуации многие элементы (включая стены) имеют ненулевую насыщенность, и inpaint размывает их
- Результат: после этих двух шагов изображение настолько изменено, что Otsu выбирает неправильный порог бинаризации

### 2. CRITICAL: auto_crop_suggest() применяется без контроля пользователя
- Если пользователь не задал crop, `auto_crop_suggest()` автоматически обрезает изображение
- Алгоритм ищет контур с area > 20% от изображения — на сложных планах может выбрать неправильную область
- Старый пайплайн НЕ делал авто-кроп без запроса пользователя

### 3. MEDIUM: Порядок операций отличается от рабочего
- Старый (рабочий): rotate → crop → grayscale → GaussianBlur → Otsu+invert → morph_close → noise_removal
- Новый (сломанный): rotate → CLAHE → color_filter+inpaint → auto_crop → grayscale → Otsu → morph_close+open → invert_if_needed → text_detect → text_remove+inpaint
- Ключевое отличие: старый пайплайн сразу шёл в grayscale+Otsu, новый добавляет 3 тяжёлых шага перед бинаризацией

### 4. LOW: invert_if_needed() может работать неправильно после новых преобразований
- После CLAHE + color_filter + inpaint соотношение белых/чёрных пикселей меняется
- Функция может инвертировать маску когда не нужно

### 5. ОТСУТСТВУЕТ: Фронтенд не обновлён
- Нет интерфейса для просмотра/редактирования vectorization data (комнаты, двери, текст)
- Нет отображения обнаруженных комнат поверх маски
- Нет возможности настроить параметры пайплайна (пороги, включение/выключение шагов)

## Что нужно исправить

### Приоритет 1: Восстановить качество маски
1. **Сделать шаги 1-3 опциональными** — normalize_brightness, color_filter, auto_crop_suggest НЕ должны применяться по умолчанию. Они должны быть отключены, пока не будут отлажены на реальных планах.
2. **Вернуть рабочий порядок бинаризации** — grayscale → GaussianBlur → Otsu+invert → morphology → noise removal. Это работало.
3. **Параметры по умолчанию** — `saturation_threshold=50` слишком низкий для планов эвакуации. Нужно поднять до 100-120 или сделать настраиваемым.
4. **auto_crop_suggest() не должен применяться автоматически** — только если пользователь явно нажал "Кадрировать". Сейчас если `crop is None`, вызывается `auto_crop_suggest()` — это неправильно.

### Приоритет 2: Тестирование на реальных данных
5. **Добавить визуальные тесты** — сохранять промежуточные результаты каждого шага (после CLAHE, после color_filter, после crop, после binarize) для отладки.
6. **Сравнить со старым пайплайном** — на тех же входных изображениях новый пайплайн должен давать результат не хуже старого.

### Приоритет 3: Фронтенд (отдельный тикет)
7. Отображение vectorization data (комнаты, двери) — это отдельная фича (floor-editor), не блокирует исправление маски.

## Файлы для изменения

- `backend/app/services/mask_service.py` — основной файл, порядок вызовов пайплайна
- `backend/app/processing/pipeline.py` — параметры по умолчанию функций
- `backend/app/api/reconstruction.py` — возможно, добавить параметры в запрос

## Как воспроизвести

1. Загрузить план эвакуации
2. Нажать "Автоматический расчёт" на шаге 2 (Маска стен)
3. Увидеть чёрную маску с мелкими белыми точками вместо стен

## Ожидаемый результат

Маска должна показывать чёткие белые стены на чёрном фоне, как было до vectorization-pipeline.
