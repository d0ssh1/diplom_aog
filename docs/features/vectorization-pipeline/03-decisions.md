# Design Decisions: Vectorization Pipeline

## Decisions

| # | Decision | Choice | Alternatives | Rationale |
|---|----------|--------|--------------|-----------|
| 1 | Where to integrate BinarizationService and ContourService | Refactor existing MaskService and ReconstructionService to call them | Create new VectorizationService that wraps both | Minimal disruption to existing API. MaskService already orchestrates preprocessing, ReconstructionService already orchestrates mesh building. Adding new service would duplicate orchestration logic. |
| 2 | How to store VectorizationResult | JSON column in Reconstruction table | Separate vectorization_results table with FK | Simpler schema, no JOIN needed. VectorizationResult is always 1:1 with Reconstruction. JSON column supported by both SQLite (TEXT) and PostgreSQL (JSONB). Downside: cannot query inside JSON easily, but we don't need to (always load full result). |
| 3 | Color filtering method | HSV saturation mask + cv2.inpaint | RGB threshold + morphology | HSV separates color from brightness, more robust to lighting variations. Saturation channel directly measures "colorfulness". Inpaint fills removed regions smoothly. RGB threshold would require separate thresholds for green/red, less generalizable. |
| 4 | Auto-crop strategy | Suggest crop, user confirms | Auto-apply crop | User may want to include legend or adjust boundary. Auto-suggest provides good default while preserving control. Existing CropSelector component already supports this UX pattern (frontend/src/components/CropSelector.tsx). |
| 5 | Adaptive binarization selection | Histogram analysis: bimodal → Otsu, else → adaptive | Always use adaptive | Otsu is faster and more accurate for scans with clear bimodal histogram (most evacuation plans). Adaptive is slower but handles uneven lighting (phone photos). Automatic selection gives best of both. Implementation: cv2.calcHist → check for two peaks. |
| 6 | Text detection library | pytesseract (Tesseract OCR) | EasyOCR, PaddleOCR | pytesseract already installed (requirements.txt), mature, supports Russian + English. EasyOCR requires PyTorch (large dependency). PaddleOCR requires PaddlePaddle (large dependency). Tesseract sufficient for printed text on evacuation plans. |
| 7 | Room number pattern | Regex: `^\d{3,4}[А-Яа-яA-Za-z]?$` or `^[A-ZА-Я]\d{3,4}$` | ML classifier | Evacuation plans follow GOST standard: room numbers are 3-4 digits optionally followed by letter, or letter followed by 3-4 digits. Regex is deterministic, fast, no training needed. ML would be overkill. |
| 8 | Room detection method | Invert mask + connected components | Contour hierarchy (RETR_TREE) | Inversion is simpler and more robust. Walls become background (0), rooms become foreground (255). Connected components directly gives room polygons. Contour hierarchy requires parsing parent-child relationships, more complex. Research (contours.py:45-73) shows RETR_TREE already implemented but not used for room detection. |
| 9 | Corridor classification | Aspect ratio > 3:1 → corridor | Width < threshold → corridor | Aspect ratio is scale-invariant, works for any image size. Width threshold would require knowing scale factor first (chicken-egg problem). 3:1 ratio empirically separates corridors (long narrow) from rooms (roughly square). Can be tuned per building type if needed. |
| 10 | Door detection method | Dilate walls, find gaps between rooms | Hough lines for door symbols | Dilation-based method is geometric, works even if door symbols are removed/unclear. Hough lines would detect door symbols (rectangles) but evacuation plans often omit them or use inconsistent symbols. Gap detection is more reliable. |
| 11 | Wall thickness computation | Distance transform → median | Contour width → mean | Distance transform gives pixel-wise distance to nearest background pixel. Median is robust to outliers (corners, junctions). Mean would be skewed by thick junctions. Implementation: cv2.distanceTransform(mask, DIST_L2, 5) → np.median(nonzero values). |
| 12 | Scale factor estimation | wall_thickness_px / 0.2m (standard wall) | User input | Automatic estimation provides reasonable default. Standard interior wall ≈ 0.2m (GOST 21.501-2018). User can override in floor-editor if needed. Better than no scale (pathfinding needs meters for time estimation). |
| 13 | Coordinate normalization | [0, 1] relative to cropped area | Pixel coordinates | Normalized coordinates are resolution-independent, easier for frontend rendering (multiply by canvas size). Building-assembly can stitch sections by aligning normalized coords. Pixel coords would require tracking original image size everywhere. |
| 14 | Processing layer architecture | Pure functions in pipeline.py, service classes in binarization.py/contours.py | Convert service classes to pure functions | Existing BinarizationService and ContourService are already implemented and tested (research shows they exist). Refactoring them to pure functions would be large effort with no benefit. Hybrid approach: keep service classes, add pure functions for new steps (color_filter, auto_crop, etc.). Violates architecture.md slightly but pragmatic. |
| 15 | API endpoint design | Add GET/PUT /reconstructions/{id}/vectors | Embed in existing GET /reconstructions/{id} | Separate endpoint keeps responses focused. GET /reconstructions returns metadata (id, name, status, url). GET /vectors returns full VectorizationResult (can be large JSON). Avoids bloating main endpoint. Follows REST principle: one resource per endpoint. |
| 16 | Backward compatibility | Keep existing endpoints unchanged, add new ones | Modify existing POST /reconstructions to return VectorizationResult | Existing frontend expects CalculateMeshResponse shape. Changing it would break frontend. Adding new endpoint is safer. Old reconstructions (before this feature) have vectorization_data=NULL, handled gracefully. |
| 17 | Error handling strategy | Update DB status=ERROR, save error_message, return 500 | Retry failed steps automatically | Retry would complicate orchestration (which step failed? how many retries?). Most errors are deterministic (bad image, no contours). Better to fail fast, show error to user, let them fix input (re-upload, adjust crop). Status=ERROR allows user to see which reconstructions failed. |
| 18 | Text removal method | cv2.inpaint with INPAINT_TELEA | Mask out text regions (set to background color) | Inpaint fills removed regions by propagating surrounding pixels, creates smoother result. Masking would leave rectangular holes that might be detected as rooms. INPAINT_TELEA is fast, good for text-sized regions. |
| 19 | Frontend changes | Minimal: add overlay to show detected rooms/doors | Full interactive editor | Interactive editor (click rooms, edit polygons) is separate feature (floor-editor). This feature focuses on backend pipeline. Frontend only needs to display results for verification. Overlay shows: walls (white lines), rooms (colored polygons), room numbers (text labels). |
| 20 | Migration strategy | Add vectorization_data column as nullable | Require backfill for old reconstructions | Nullable allows gradual migration. Old reconstructions work as before (3D mesh only). New reconstructions get VectorizationResult. Backfill not feasible (original masks may be deleted, processing parameters unknown). |

## Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Tesseract OCR fails to detect room numbers | Medium | Graceful degradation: room.name remains empty, admin fills manually in floor-editor. System continues without room numbers. Test with plans 2 and 3 (no numbers) to ensure robustness. |
| Auto-crop detects wrong boundary (includes legend) | Low | User can adjust crop manually via CropSelector. Auto-crop is suggestion, not forced. Fallback: if no boundary detected, use full image. |
| Color filtering removes actual walls (if walls are colored) | Low | Evacuation plans follow GOST: walls are black/gray (achromatic). Colored elements are always annotations (arrows, symbols). Saturation threshold tuned empirically on test plans. If issue arises, make threshold configurable. |
| Room detection creates too many small rooms (noise) | Medium | Filter by area: rooms < min_area (e.g. 1000px²) discarded as noise. Threshold tuned on test plans. Admin can merge/delete rooms in floor-editor. |
| Door detection misses doors or creates false positives | Medium | Dilation kernel size tuned empirically. False positives (gaps that aren't doors) can be deleted in floor-editor. Missing doors can be added manually. Door detection is best-effort, not critical for 3D mesh. |
| Wall thickness estimation is inaccurate | Low | Affects scale factor only. Pathfinding uses scale for time estimation (not critical). Building-assembly uses normalized coords (not affected). If inaccurate, admin can override scale in floor-editor settings. |
| VectorizationResult JSON becomes too large (>1MB) | Low | Typical plan: ~50 walls, ~20 rooms, ~10 doors, ~30 text blocks. Each entity ~100 bytes. Total ~10KB. Even complex plans <100KB. PostgreSQL JSONB handles MB-sized JSON efficiently. SQLite TEXT column has no practical limit. |
| Processing takes too long (>30s) | Medium | Most steps are fast (<1s). Slowest: text detection (pytesseract ~2-5s), room detection (~1-2s). Total ~10-15s for typical plan. If timeout, increase FastAPI timeout or make endpoint async with status polling. Current implementation already async (ReconstructionService.build_mesh is async). |
| Coordinate normalization loses precision | Low | Float64 has 15 decimal digits. [0,1] range with 15 digits = sub-pixel precision even for 8K images. Sufficient for all use cases. |
| Hybrid architecture (service classes + pure functions) confuses implementers | Medium | Document clearly in 01-architecture.md and phase plans. Existing code (BinarizationService, ContourService) stays as-is. New code (pipeline.py) uses pure functions. Implementers follow phase plan, no ambiguity. |

## Open Questions

- [x] **Should we support automatic rotation detection?** → NO. User rotates manually via button (90° increments). Automatic rotation detection (via Hough lines or text orientation) is complex, error-prone, and not needed (user can see if plan is rotated). Existing UI already has rotation button.

- [x] **Should we detect stairs/elevators automatically?** → NO. Ticket explicitly says "НЕ детектить лестницы/лифты автоматически (админ пометит в floor-editor)". Stairs/elevators have varied visual representations, hard to detect reliably. Admin marks them in floor-editor.

- [x] **Should we support multi-page plans (building with multiple sections)?** → NO, not in this feature. Building-assembly is separate feature. This feature processes one plan image → one VectorizationResult. Building-assembly will load multiple VectorizationResults and stitch them.

- [x] **Should we validate that room numbers are unique?** → NO. Duplicate room numbers are valid (e.g. different floors, different buildings). Uniqueness enforced at Building level, not FloorPlan level. This feature only extracts numbers, doesn't validate.

- [x] **Should we support plans with multiple floors in one image?** → NO. One image = one floor. If plan shows multiple floors (rare), user crops each floor separately and uploads as separate reconstructions.

- [x] **Should we cache intermediate results (e.g. binary mask before text removal)?** → NO. Processing is fast enough (<15s), caching adds complexity. If user wants to retry with different parameters, they re-upload. Intermediate results not exposed in API.

- [x] **Should we support undo/redo for vectorization edits?** → NO, not in this feature. Floor-editor will handle undo/redo for manual edits. This feature only produces initial VectorizationResult.

- [ ] **What if pytesseract is not installed on deployment server?** → Make pytesseract optional dependency. If not installed, skip text detection step (text_blocks = []), log warning. System continues without room numbers. Document in deployment guide.

- [ ] **Should we expose intermediate pipeline steps in API (e.g. GET /masks/{id}/color-filtered)?** → Defer to implementation. Not required for MVP. Could be useful for debugging, but adds API surface. If needed, add in future iteration.
