# Design Decisions: Building Hierarchy

## Decisions

| # | Decision | Choice | Alternatives | Rationale |
|---|----------|--------|--------------|-----------|
| 1 | Где живёт привязка plan↔section | `Section.reconstruction_id` (FK на reconstructions, nullable, UNIQUE) | (a) `Reconstruction.section_id`; (b) join-таблица | Семантически Section «владеет» планом и решает, какой mesh показать. Reconstruction может существовать без секции («висящий план»). 1:1 — join-таблица избыточна. UNIQUE-constraint обеспечивает 1:1. |
| 2 | Старые reconstructions | Полностью удалить миграцией | (a) Оставить с floor_id=NULL; (b) bulk-привязать к "default" корпусу | Пользователь подтвердил: "Все реконструкции не имеют ценности сейчас". Удаление избегает мёртвых записей и упрощает миграцию. |
| 3 | Имя поля корпуса | `Building.code` (1-5 заглавных букв, UNIQUE) + сохранить `Building.name` | (a) только `name`; (b) только `code` | Селектор показывает буквы (S/D/B), но в админке нужно человекочитаемое имя ("Корпус D"). Разделение полей даёт обе возможности. |
| 4 | Где валидировать обязательность building+floor в wizard | На клиенте — кнопка Next disabled; на сервере — `floor_id: int` (не Optional) в SaveRequest | (a) Только клиент; (b) Только сервер | Клиент даёт UX-сигнал; сервер защищает от прямых вызовов API. Дублирование валидации стандартно для FastAPI/Pydantic. |
| ~~5~~ | ~~Хранение фото-схемы этажа~~ | **ОТМЕНЕНО (см. ADR-25):** хранить как UploadedFile + ссылка из `Floor.schema_image_id` | — | См. ADR-25. Multi-step wizard требует возвращаться к схеме между сессиями + использовать на end-user UI как фон мини-карты. |
| 6 | Координаты секций | Нормализованные [0,1] | Абсолютные пиксели | Согласуется с правилом «All coordinates after vectorization normalized to [0, 1]» (CLAUDE.md). Делает мини-карту resolution-independent. |
| 7 | Стратегия сохранения секций этажа | "Replace all": атомарно удалить все + вставить новые в одной транзакции | Diff create/update/delete по id | Редактор работает с целым этажом (≤20 секций), полная замена проще, без оптимистических конфликтов. |
| 8 | Геометрия секции в БД | JSON-колонка `{type: 'rect'\|'polygon', points: [[x,y],...]}` | (a) Отдельные таблицы; (b) WKT/GeoJSON | Простая полу-структурированная форма; рендерится в SVG. SQLite 3.45+ и PostgreSQL поддерживают JSON нативно. Геопространственные индексы не нужны. |
| 9 | Section.section_type | Enum-int с дефолтом `1=room`, зарезервированы `2=stairs`, `3=elevator` | Не добавлять поле сейчас | "Архитектура должна допускать лестницы/лифты в будущем". Поле резервирует возможность без миграции. На UI пока скрыто. |
| 10 | Отдельный экран `/admin/buildings` | Да, новая страница | (a) Встроить в DashboardPage; (b) Внутри FloorEditor | Корпуса и этажи — самостоятельная сущность; чистый разделитель ответственности. |
| 11 | URL `/admin/floor-editor` без id | Выбор корпуса+этажа внутри страницы | `/admin/floor-editor/:floorId` | Админ часто переключается между этажами; убирает ненужную навигацию. Опционально query `?floor=101` для глубоких ссылок. |
| 12 | Видимость "висящих" реконструкций для end-user | Скрыты | Показывать с пометкой "не привязан" | Пользовательское ТЗ: "Без отсеков нельзя". Защита от показа неполных корпусов. |
| 13 | Связь UC-07 (multifloor route) с секциями | Не меняем backend маршрутизации; маппинг segment→section на клиенте | Включить section в `MultifloorRouteResponse` | Out of scope. Существующий API не ломаем. Маппинг на клиенте достаточен для подсветки. |
| 14 | Старые поля `Reconstruction.building_id`/`floor_number` | Удалить миграцией (вместе с дропом старых записей) | Сохранить для совместимости | Поля заменяются на `floor_id` (FK). После миграции данных нет. |
| 15 | Совместимость с FloorTransition | Дроп таблицы `floor_transitions` в миграции; переcоздание после | Перенести `transition.building_id` → `transition.from_floor_id` | Out of scope. После dropа всех reconstructions старые транзишены теряют валидность. Документировать в README. |
| 16 | Имя нового публичного экрана | `FloorViewerPage` на `/viewer` | Объединить с `RouteTestPage` | `RouteTestPage` — тестовый экран в `/admin`. Публичный — отдельная страница. |
| ~~17~~ | ~~Диалог BindReconstruction~~ | **ОТМЕНЕНО (см. ADR-30):** галерея с фильтрами Здание+Этаж, default «Все здания / Все этажи» | — | См. ADR-30. Мокап показывает галерею карточек, не dropdown. |
| 18 | Auth для `GET /buildings?published=true` | Логин-обязателен (user role, не admin) | (a) Public без JWT; (b) Только admin | В системе есть две роли: admin и обычный авторизованный user. End-user экран — для второй. PublicHomePage даёт вход; после логина — переход на `/viewer`. Реализация — `Depends(get_current_user)` (любая роль) внутри ветки `published=true`; для admin-ветки `Depends(get_current_admin_user)`. |
| 19 | Mesh-форматы в `PublicBuilding` | Только `mesh_url_glb` | Оба `mesh_url_glb` + `mesh_url_obj` | End-user рендерит через Three.js (GLTFLoader) — нужен только GLB. OBJ остаётся доступен в admin-API. Если в будущем понадобится OBJ на публичном экране — добавить полем без breaking change. |
| 20 | `Section.section_type` диапазон в Pydantic | `ge=1, le=10` | Strict enum 1\|2\|3 | Резерв для будущих типов (телепорты, пожарные выходы). Текущее именование (1=room, 2=stairs, 3=elevator) — в коде/комментарии. |
| 21 | Иерархическая published-фильтрация | Корпус виден ⇔ ≥ 1 этаж виден ⇔ ≥ 1 секция с reconstruction.status=Done | Только building-level фильтр | Если у корпуса этаж 8 пустой, а этаж 7 заполнен — на end-user в селекторе этажей показывается только 7. Иначе пользователь увидит «пустой» этаж 8 без секций. Реализация — фильтрация при сборке `PublicBuilding` payload в `BuildingService.list_published`. |
| 22 | UI-порядок селектора Корпус→Отсек→Этаж (как на макете) vs data-зависимость B→F→S | Визуально как в макете, но при смене этажа section-список перестраивается под этажа | Упорядочить визуально как Корпус→Этаж→Отсек | Соответствие референс-макету. При смене Floor: если активный Section.number есть на новом этаже — оставляем; иначе берём первый доступный. Логика — в `useFloorViewer`. |
| 23 | Терминология меню (label) | "Редактор отсеков" (на UI), URL `/admin/floor-editor` (data-уровневое имя) | Только "Редактор схемы этажа" | Пользователь использует «отсек» как ведущий термин. URL остаётся data-ориентированным (этаж — родитель секций). Метка кнопки в навигации и заголовок страницы — "Редактор отсеков". |
| 24 | Когда отправляем floor_id на бэк | На шаге StepUpload, когда пользователь выбрал building+floor — сразу обновляем reconstruction (PATCH /reconstructions/{id}) | На последнем шаге StepSave | Если admin прервёт wizard на середине, plan уже привязан к этажу. PATCH-эндпоинт лёгкий. |
| 25 | FloorEditor — single-page vs multi-step wizard | **Multi-step wizard 1→2→3→4→5** + Overview (graphical/table) + Context Menu | Single-page editor со всеми инструментами одновременно | Соответствие референс-макету. Линейный flow упрощает обучение, разгружает UI. Wizard ↔ Overview переключается в зависимости от состояния этажа. |
| 26 | Хранение фото-схемы этажа (ОТМЕНЯЕТ ADR-5) | `Floor.schema_image_id` (FK→uploaded_files) | Не хранить | Wizard возвращается к схеме между сессиями. End-user mini-map использует как фон. Альтернатива (re-upload каждый раз) ломает UX. |
| 27 | Хранение wall_polygons | `Floor.wall_polygons: JSON` (нормализованные [0,1] полигоны) | Регенерация из изображения при каждом запросе | Шаг 3 даёт результат смешанный (CV+ручная правка) — результат руки не воспроизводим. Сохраняем финальный артефакт. Используется как фон у end-user'a. |
| 28 | Геометрия секции (УПРОЩАЕТ ADR-8) | 4-точечный полигон (повёрнутый прямоугольник): `{points: [[x,y]×4]}`, без discriminator type | Discriminated union rect/polygon | Мокап показывает только rect-инструмент. 4-точечный quad = axis-aligned rect (дегенеративный случай) ИЛИ повёрнутый. Полигон 5+ точек не нужен для отсеков. |
| 29 | Поля Section.description, Section.color | НЕ добавляем | Как в первоначальном дизайне с description+color | Пользователь явно отказался от обоих. Цвет фиксированный (orange = active, neutral = idle). Описание считаем избыточным. |
| 30 | Plan binding UI | **Галерея карточек** + поиск + dropdown'ы Здание/Этаж (default «Все»). Показываются ВСЕ Done-реконструкции, не только этого этажа (ОТМЕНЯЕТ ADR-17) | Dropdown «висящие планы этого этажа» | Пользователь явно подтвердил: «можно будет выбрать здание и этаж... изначально все здания, все этажи». Это даёт гибкость привязать план любого этажа (полезно при переразметке/корректировке). Защита от ошибок переходит на UI-фильтры. |
| 31 | Wall extraction на шаге 3 | Авто (существующий CV pipeline: binarization → contours → vectorizer) **+ ручная правка** через те же tools | Только автомат / Только вручную | Пользователь подтвердил: «Автоматический, как уже есть при загрузке + ручная загрузка». Реюз `processing/binarization.py`, `processing/contours.py`, `processing/vectorizer.py`. Вызов через новый `FloorSchemaService.extract_walls()`. |
| 32 | Кнопка в Overview (шаг 6) | «Сохранить изменения» (orange) | «Редактировать» (как нарисовано в макете) | Пользователь поправил: «Это ошибка — кнопка должна быть не редактировать, а сохранить изменения». Шаг 7 — context menu в overview, не отдельный экран. |

## Risks

| ID | Risk | Impact | Mitigation |
|----|------|--------|------------|
| R-1 | Миграция дропает все reconstructions — потеря активной работы | High | Согласовано с пользователем. В dev — без последствий. Перед prod — бэкап. |
| R-2 | Нумерация секций не уникальна между этажами одного корпуса — путаница в UI | Med | UI всегда показывает иерархию "Корпус → Этаж → Отсек"; `displayLabel` включает путь. UNIQUE-constraint только на (floor_id, number). |
| R-3 | После drop reconstructions старые `floor_transitions` указывают на несуществующие id | Med | Миграция: сначала `DELETE FROM floor_transitions`, потом drop+recreate reconstructions. Транзишены админ переcоздаёт после миграции. |
| R-4 | Полигон с самопересечением сломает рендер мини-карты | Low | Клиентская валидация при создании. Сервер сохраняет «как есть». |
| R-5 | Десинхронизация UI: админ редактирует секции, другой админ удаляет этаж | Low | Replace-стратегия с проверкой `floor_id` → 404 → клиент перезагружает каталог. |
| R-6 | Производительность мини-карты при сложных полигонах | Low | SVG; для ≤20 секций × ≤30 точек overhead незначительный. |
| R-7 | Каскадное удаление Building уносит чужие реконструкции | Med | Каскад только на Floor→Section; `Reconstruction.floor_id ON DELETE SET NULL`. План «отвязывается», не теряется. |
| R-8 | Существующий `RouteTestPage` использует все reconstructions без иерархии | High | Адаптировать `useRouteTest` под новую схему: фильтровать по floor_id, метки строить из Building.code + Floor.number + Section.number. Включается в scope. |
| R-9 | Если admin загрузил план, заполнил building+floor, но wizard прервался — реконструкция остаётся «полу-готовой» (status≠Done) с floor_id | Low | Висящий план виден в `/admin/floor-editor` в списке unbound только если `status=Done`. Полу-готовые скрыты. Существующая reconstructions-страница (admin) их видит и админ может вручную удалить. |
| R-10 | `floor_transitions` сейчас имеет FK на reconstructions — drop reconstructions без предварительного DROP transitions невозможен (FK constraint) | High | В Alembic-миграции явный порядок: `DROP TABLE floor_transitions` → `DROP TABLE rooms` (тоже FK) → `DROP TABLE reconstructions` → пересоздание `reconstructions` (с FK floor_id) → пересоздание `floor_transitions` (со старой схемой) → новые таблицы `sections`. Документировать в Phase 1 кодпланa. |
| R-11 | Хранение фото-схем этажей увеличит нагрузку на storage (≥1 файл/этаж × N этажей × N корпусов) | Low | Один файл = ~1-5MB. Для типичного университета (~5 корпусов × 5 этажей = 25 файлов = ≤125MB) приемлемо. Cleanup при `DELETE Floor` — каскад на UploadedFile через FK ON DELETE SET NULL не уничтожает file (это на phase реализации решит admin). |
| R-12 | Wall extraction для большого изображения (10000×10000+) может занимать > 30 секунд | Med | На клиенте показывать spinner с progress «Обработка стен...». Backend — синхронный вызов (не background task в первой версии). Если станет проблемой — переехать в Celery в backlog. |
| R-13 | Plan gallery с >100 реконструкциями: тормоза в UI без virtualization | Low | Реалистично ≤30 реконструкций; pagination не делаем. Если станет проблемой — добавить `?limit=50&offset=` в `/reconstructions`. |

## Open Questions

- [x] Q-1: Есть ли `Building.code` уже в БД? — Нет (`backend/app/db/models/building.py`). Добавляется новой миграцией.
- [x] Q-2: Где живёт публичный селектор — на `/` или `/viewer`? — `/viewer` (новый); `/` остаётся PublicHomePage. Финальное решение — на frontend phase.
- [x] Q-3: Полигон или прямоугольник в первой версии? — Оба сразу. Прямоугольник — частный случай полигона из 4 точек, но в JSON хранится отдельно (`type: 'rect'`) для облегчённой логики drag-edit углов.
- [x] Q-4: Должна ли мини-карта рендерить план каждой секции (превью) или только нумерованный полигон? — Только нумерованный полигон + подсветка.
- [ ] Q-5: Импорт конфигурации корпуса из JSON-файла? — Out of scope.
- [x] Q-6: Что при удалении секции с активными транзишенами? — Транзишены ссылаются на reconstruction, не на section. Отвязка от секции не ломает транзишены. Удаление reconstruction каскадирует на транзишены (существующее поведение).
- [x] Q-7: Адаптация `RouteTestPage` под новую схему — отдельный тикет или часть фичи? — Включаем в эту фичу как обязательную часть (R-8), иначе сломается после миграции.
- [ ] Q-8: Как пользователь возвращается к списку корпусов? — Кнопка «← ДВФУ» в шапке (как на макете). Имя «ДВФУ» — литерал на фронте (нет понятия «комплекс» в модели).
- [ ] Q-9: Wishlist (отложено): отображение начальной/конечной точки маршрута на 3D mesh активного отсека — пользователь упомянул как «было бы здорово, но сложно». Технически это маркеры в Three.js scene по координатам точек из `MultifloorRouteResponse`. Откладываем в backlog как отдельный тикет.
- [ ] Q-10: Сценарий «отвязать план от секции» — отдельная кнопка в FloorEditor (нажал на привязанную секцию → диалог → «Не привязывать»)? — Да, диалог BindReconstructionDialog имеет вариант «Не привязывать» (см. UC-04.3). Это эквивалентно сохранению секции с `reconstruction_id=null`.
- [ ] Q-11: Что если пользователь введёт «D304» — это `D` корпус + `304` номер комнаты? Или `D3` корпус + `04` секция? — Парсинг: 1-N начальных букв = building.code; остаток = room number. То же поведение, что в текущей системе. Не меняем.
