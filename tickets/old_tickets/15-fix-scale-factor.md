## Проблема

Коридорная маска теперь правильная (тикет 23 исправлен), скелет идёт по внутренним коридорам, двери привязываются корректно (snap ~42-46px). Но маршрут в 3D **всё ещё смещён** — бирюзовая линия сдвинута и выходит за пределы стен.

### Причина

Сохранённый `_nav.json` содержит `"scale_factor": 0.05`, а 3D-меш строится с `scale = 0.02` (`pixels_per_meter=50`). Все координаты маршрута получаются **в 2.5 раза** больше, чем нужно и **вылетают за bounding box** модели.

### Доказательство (числа из текущего nav.json)

Маска: 863×368. Bounding box 3D-модели (scale=0.02): **x=[0, 17.26], z=[−7.36, 0]**

| Узел | 2D pixel | 3D с 0.05 (текущее) | 3D с 0.02 (правильное) | В пределах меша? |
|------|----------|---------------------|------------------------|-----------------|
| Комната 1 | (585.9, 139.4) | (29.30, −11.43) | (11.72, −4.57) | 0.05: **НЕТ** / 0.02: ДА |
| Дверь 1 | (567.4, 146.8) | (28.37, −11.06) | (11.35, −4.42) | 0.05: **НЕТ** / 0.02: ДА |
| Entry 1 | (524.0, 163.0) | (26.20, −10.25) | (10.48, −4.10) | 0.05: **НЕТ** / 0.02: ДА |
| Entry 2 | (155.0, 107.0) | (7.75, −13.05) | (3.10, −5.22) | 0.05: **НЕТ** / 0.02: ДА |
| Комната 2 | (180.7, 91.5) | (9.04, −13.82) | (3.61, −5.53) | 0.05: **НЕТ** / 0.02: ДА |

С `scale_factor=0.05` x-координаты до 29.30 при максимуме модели 17.26 — маршрут на **70% правее** границы модели.

## Что нужно сделать

### Исправление 1: Найти и исправить РЕАЛЬНЫЙ источник scale_factor=0.05

**Внимание:** Сигнатура `build_graph()` в `nav_service.py` УЖЕ содержит `scale_factor: float = 0.02` (строка 44), но сгенерированный JSON всё равно записывает `0.05`. Это значит что где-то при вызове значение **передаётся явно**.

Выполнить поиск:
```bash
grep -rn "0\.05" backend/app/
grep -rn "build_graph" backend/app/api/
grep -rn "scale_factor" backend/app/
```

Типичные места где может быть `0.05`:
- `backend/app/api/reconstruction.py` — вызов `build_graph(..., scale_factor=0.05)`
- Другой сервис, который оборачивает вызов
- Переменная/константа в конфиге

**Везде** где `scale_factor` связан с навигацией/графом — заменить `0.05` на `0.02`.

### Исправление 2: Дефолт scale_factor в find_route()

**Файл:** `backend/app/services/nav_service.py`
**Метод:** `find_route()` — строка с `metadata.get('scale_factor', ...)`

```python
# БЫЛО:
scale_factor = metadata.get('scale_factor', 0.05)
# СТАЛО:
scale_factor = metadata.get('scale_factor', 0.02)
```

### Как найти ВСЕ места

Текущий `nav_service.py` показывает `build_graph(scale_factor=0.02)` в сигнатуре (строка 44), но сгенерированный JSON всё равно содержит `0.05`. Это значит что `0.05` передаётся **явно** при вызове из API endpoint.

```bash
# 1. Найти ВСЕ упоминания 0.05 в бэкенде:
grep -rn "0\.05" backend/app/

# 2. Найти все вызовы build_graph с аргументами:
grep -rn "build_graph" backend/app/api/
grep -rn "build_graph" backend/app/services/

# 3. Найти все упоминания scale_factor:
grep -rn "scale_factor" backend/app/
```

**Вероятные места где передаётся 0.05:**
- `backend/app/api/reconstruction.py` — API endpoint может вызывать `nav_service.build_graph(mask_id, rooms, doors, scale_factor=0.05)`
- `backend/app/services/reconstruction_service.py` — если build_graph вызывается оттуда
- `nav_service.py` строка 127 — фоллбэк в `find_route`

**Заменить 0.05 → 0.02 ВЕЗДЕ** где это связано с `scale_factor` навигации.

---

## Откуда берётся 0.02

`mesh_builder.py` использует `pixels_per_meter=50.0`. Это означает 50 пикселей = 1 метр, то есть 1 пиксель = 1/50 = **0.02** метра. `transform_2d_to_3d()` должен использовать тот же масштаб для совпадения координат.
