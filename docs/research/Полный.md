# Research: Полный обзор кодовой базы Diplom3D

date: 2026-04-16

## Summary

Diplom3D на текущий момент — это уже работающий проект с полным основным сценарием: загрузка планов, обработка изображений, построение векторного представления, генерация 3D-модели, просмотр модели в браузере, редактирование планов и stitching нескольких источников. Backend содержит отдельные роутеры для auth, upload, reconstruction, navigation и stitching, а также сервисы, репозитории, processing-модули и тесты по основным областям.

При этом кодовая база не полностью совпадает с целевой архитектурой из `prompts/architecture.md`. Часть бизнес-логики всё ещё находится в API-роутерах и сервисах, некоторые endpoints остаются заглушками, а часть потоков на frontend использует временные допущения, касты и дублирование. Документ ниже фиксирует именно фактическое состояние кода: что уже готово, что работает частично, что не реализовано и где присутствует технический долг.

#### Data flow
1. Frontend uploads a plan through the wizard or upload components.
2. `upload.py` validates the file, stores it via `FileStorage`, writes `UploadedFile` in DB.
3. `WizardPage` calls `calculateMask`, `saveMaskAndAnnotations`, `buildNavGraph`, `buildMesh`, `save` in sequence.
4. `ReconstructionService.build_mesh()` creates reconstruction, loads mask, extracts elements, detects rooms and doors, normalizes coordinates, stores vectorization data, builds and exports mesh.
5. `navigation.py` and `nav_service.py` provide route/search behavior; API navigation endpoint still contains stubbed responses.
6. `EditPlanPage` loads reconstruction + vectors, edits rooms/doors, rebuilds nav graph, and returns to 3D or admin flow.
7. `ViewMeshPage` and `StepView3D` show the 3D mesh and route UI.

#### Error handling
- `upload.py` validates file extensions and raises 400 for unsupported formats.
- `reconstruction.py` converts service errors to HTTP exceptions and returns 404 for not found cases.
- `stitching.py` converts `ValueError` to 400/404 and other errors to 500.
- `navigation.py` currently returns placeholder route/graph data rather than a fully wired service result.

## Integration Points

### Database

- `Reconstruction` and `UploadedFile` are the primary persistence entities for the upload/reconstruction flow.
- `Building` and `Floor` models exist in both ORM and Pydantic form; they are only partially wired into the main flow.
- `vectorization_data` is stored as JSON/Text inside the reconstruction record rather than fully normalized relational tables.
- Session setup is handled by `backend/app/core/database.py`; dependency providers live in `backend/app/api/deps.py`.
- Alembic migrations are stored in `backend/alembic/versions/`.

### File storage

- `backend/app/core/config.py:26-29` defines `UPLOAD_DIR`, size limit, and allowed extensions.
- `backend/main.py:27-30` mounts uploads as static files at `/api/v1/uploads`.
- `backend/app/api/upload.py:53-66`, `90-103`, `127-140` save plans, masks, and environment photos into subfolders.
- `backend/app/services/reconstruction_service.py:235-244` saves OBJ/GLB mesh files via `FileStorage`.
- URLs for uploaded files are stored in DB and returned to frontend as preview/original links.

### API boundaries

- Backend API is mounted under `/api/v1` in `backend/main.py:41-42`.
- Auth uses bearer tokens; frontend stores JWT in `localStorage.auth_token`.
- `frontend/src/api/apiService.ts` is the single HTTP client entrypoint.
- Some endpoints are still placeholder or partially implemented, so frontend code sometimes casts API responses or handles fallback shapes.

### Pipeline

- `ReconstructionService.build_mesh()` is the main trigger for the reconstruction chain.
- `vectorizer.py` and `pipeline.py` provide the image-processing substeps: crop suggestion, text detection/removal, room detection, classification, door detection, scale factor, normalization.
- `mesh_builder.py` is responsible for the 3D mesh from mask.
- `navigation.py` handles A* and route search logic; API layer is still not fully connected to a real backend route builder.
- `stitching/` modules handle affine transform, clipping and merging for multi-plan composition.

## Notes on documentation drift

- `README.md` and `docs/ONBOARDING.md` previously described the project in a more idealized or simplified way.
- The current codebase should be documented as a **partially aligned, working system** with specific incomplete endpoints and technical debt.
- Any mention of “processing is purely functional” or “all layers are already cleanly separated” would be inaccurate for the current code.

## Technical debt and current shortcomings

- API routers still carry some business logic and placeholder flows.
- Some domain state is stored as JSON blobs instead of normalized tables.
- The reconstruction and stitching services are large and multi-responsibility.
- Frontend still uses a few casts, fallback shapes, and duplicated loading logic.
- Some flows are implemented in the UI before the backend is fully completed.
- The project has working core features, but some edge flows are stubbed or partially wired.
