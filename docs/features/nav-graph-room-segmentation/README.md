# nav-graph-room-segmentation — Design

date: 2026-03-19
status: draft

## Business Context

Текущая реализация `extract_corridor_mask` (`processing/nav_graph.py:15-141`) выделяет
коридоры через дилатацию стен с фиксированными параметрами (`kernel=7, iterations=2`).
Это работает только при определённом масштабе плана: на мелких планах дилатация
«захлопывает» коридоры вместе с дверями, на крупных — не закрывает дверные проёмы.

Замена алгоритма на `distanceTransform` свободного пространства позволяет выделять
«широкие» проходы адаптивно — порог ширины вычисляется из `wall_thickness_px`,
который уже присутствует в пайплайне (`pipeline.py:569`). Это устраняет зависимость
от масштаба и делает сегментацию коридоров надёжной для разных планов.

## Acceptance Criteria

1. `extract_corridor_mask` принимает `wall_thickness_px: float` и использует его
   для вычисления адаптивного порога вместо фиксированных `dilate_kernel_size` / `dilate_iterations`.
2. Пороговое значение: пиксели свободного пространства с `dist >= corridor_threshold`
   считаются «широким проходом» (коридором), где `corridor_threshold = k * wall_thickness_px`
   (k — настраиваемый коэффициент, default ≈ 1.5).
3. Сигнатура функции обратно совместима по поведению: возвращает `np.ndarray` той же
   формы и семантики (255 = коридор, 0 = не коридор).
4. `NavService.build_graph` передаёт уже вычисленный `wall_thickness_px` в функцию
   (без повторного вычисления).
5. Все существующие тесты проходят; добавлены новые тесты для адаптивного порога.
6. Логирование сохраняет диагностическую информацию: порог, площадь коридора.
