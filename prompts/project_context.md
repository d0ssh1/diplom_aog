Агенты ОБЯЗАНЫ прочитать этот файл при research и design фазах.

---

## Цель системы

Построение виртуальных карт зданий посредством анализа планов эвакуации.
Система принимает фотографию аварийного плана эвакуации → строит 3D-модель
этажа → даёт возможность редактировать → строит маршруты по A*.

## Роли пользователей

**Администратор:** загружает изображения планов, корректирует результат обработки,
управляет зданиями и этажами.

**Пользователь:** выбирает здание/этаж, просматривает 3D-модель, строит маршрут
между двумя точками (формат: "A304" → буква корпуса + номер помещения).

## Доменные сущности (из ВКР раздел 2.2)

### Здание (Building)
- id: уникальный идентификатор
- name: название ("Корпус А")
- floors: список этажей

### Этаж (Floor)
- id: уникальный идентификатор
- building_id: ссылка на здание
- floor_number: номер этажа (уникален в пределах здания)
- vector_model_id: ссылка на векторную модель
- navigation_graph_id: ссылка на навигационный граф

### Помещение (Room)
- location: координаты центра (x, y)
- name: строка 4-15 символов (латиница, кириллица, цифры, "-", "_")
- room_type: {помещение, лестница, лифт, эвакуационный_выход, коридор}

### Векторная модель (VectorModel / FloorPlan)
- id: уникальный идентификатор
- floor_id: ссылка на этаж
- walls: набор контуров стен (замкнутые геометрические фигуры)
- rooms: набор помещений с полигонами

### Навигационный граф (NavigationGraph)
- id: уникальный идентификатор
- floor_id: ссылка на этаж
- nodes: вершины (id, координаты x/y, тип)
- edges: рёбра (from_id, to_id, weight=евклидово расстояние)

### Маршрут (Route)
- start: (x, y, z) где z = номер этажа
- end: (x, y, z)
- path: упорядоченная последовательность рёбер графа
- total_distance: длина маршрута
- estimated_time: примерное время прохождения

## Ограничения целостности (из ВКР раздел 2.3)

### Изображение плана
- Формат: PNG или JPG
- Разрешение: ≥ 1000×1000 пикселей
- Размер: ≤ 50 МБ
- Должно быть полным, не обрезанным, не размытым

### Помещение
- Название: 4-15 символов, латиница/кириллица/цифры/"-"/"_"
- Координата: кортеж (X, Y, Z), где X,Y ∈ ℝ, Z ∈ ℤ ∩ [0, 12]

### Векторная модель
- Привязана к существующему этажу
- Элементы в пределах границ исходного плана
- Контуры помещений — замкнутые фигуры

### Навигационный граф
- Построен на основе векторной модели
- Вес рёбер > 0
- Вершины только в проходимых зонах (коридоры, помещения), НЕ внутри стен

### Маршрут
- Минимум 2 точки (начало + конец)
- Точки в пределах виртуальной карты

## Пайплайн обработки (из ВКР раздел 4.2.2)

### CM_001.1: Бинаризация
- Вход: цветное изображение плана
- Метод Оцу для автоматического порога
- Всё темнее порога → стены (255), светлее → фон (0)
- Выход: бинарное чёрно-белое изображение

### CM_001.2: Распознавание текста
- Вход: бинарное изображение
- Поиск групп мелких объектов (похожих на буквы)
- OCR (pytesseract) для распознавания
- Сохранение текста + координат
- Выход: изображение без текста + данные текстовых блоков

### CM_001.3: Выделение структурных объектов
- Вход: бинарное изображение без текста
- findContours для иерархии замкнутых контуров
- Аппроксимация (Douglas-Peucker) для упрощения
- Классификация: стены (крупные внешние контуры), двери (разрывы в стенах)
- Выход: распознанные объекты с 2D-координатами

### CM_001.4: Построение 3D-модели
- Вход: 2D-контуры объектов
- Опорные точки для матрицы преобразования пиксели→метры
- Экструзия стен на высоту этажа
- Формирование проёмов (двери, окна)
- Выход: 3D-модель (OBJ, GLB, JSON)

## Алгоритм навигации (из ВКР раздел 3.2.5)

### Построение графа (RM_001)
1. Определение проходимых зон (коридоры, помещения, двери)
2. Создание вершин:
   - В центрах помещений
   - В дверных проёмах (связь между комнатами)
   - Вдоль коридоров с шагом 2-3 метра
   - У лестниц (переход между этажами)
3. Создание рёбер: соединение соседних вершин с проверкой,
   что отрезок не пересекает стены
4. Вычисление весов: евклидово расстояние между вершинами

### Поиск пути — A* (RM_002)
- f(v) = g(v) + h(v)
- g(v) — стоимость пути от старта до v
- h(v) — эвристика (манхэттенское расстояние для сетки)
- Эвристика допустима и монотонна → A* оптимален
- Постобработка: "спрямление" пути (удаление промежуточных точек
  если между ними нет препятствий)

### Формат ввода маршрута
- "A304" → буква корпуса + номер помещения
- Система находит помещения в БД → загружает графы → ближайшие вершины → A*

## Классы эквивалентности для тестирования (из ВКР таблица 1)

### Загрузка изображения
- Правильный: PNG/JPG, ≥1000×1000, ≤50МБ → успех
- Неправильный: не PNG/JPG → ошибка формата
- Неправильный: <1000×1000 → ошибка разрешения
- Неправильный: >50МБ → ошибка размера
- Неправильный: повреждённый файл → ошибка чтения

### Бинаризация
- Правильный: высокая контрастность, чёткие границы → успех
- Неправильный: низкая контрастность, размытие, тени → ошибка

### Выделение объектов
- Правильный: чёткие стены, двери, лестницы → распознано
- Неправильный: мелкие/размытые элементы → ошибка распознавания

### Построение маршрута
- Правильный: существующие помещения, формат "A304" → маршрут
- Неправильный: несуществующие точки → ошибка
- Неправильный: изолированные области → нет пути
- Неправильный: некорректный формат ввода → ошибка формата

## Элементы плана эвакуации (онтология)

Каждый план содержит: коридоры, помещения, лифты, лестницы,
двери, текст, пожарные выходы, электрощиты.

Типы элементов карты (множество):
{помещение, лестница, лифт, эвакуационный_выход, огнетушитель,
электрощит, ручной_пожарный_извещатель, путь_к_выходу, текст}
