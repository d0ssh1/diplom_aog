# Research: Смещение кабинетов / векторной маски / аварийного плана
date: 2026-03-31

## Summary
В кодовой базе есть три основных связанных потока: кадрирование и предобработка плана, ручное редактирование маски/кабинетов на фронте, а также построение векторизации, меша и навигационного графа на бэке. Для этих потоков уже существуют роутеры, сервисы, Pydantic-модели, ORM-таблицы и фронтовые шаги wizard/editor. В текущем коде координаты и геометрия проходят через несколько слоёв: кроп и rotation на фронте, crop/rotation в `mask_service`, нормализация координат в `pipeline.py`, затем повторное использование данных в `reconstruction_service` и `nav_service`.

Связанные с багом места находятся в `frontend/src/components/Wizard/StepPreprocess.tsx:28`, `frontend/src/components/Wizard/StepWallEditor.tsx:45`, `frontend/src/components/Editor/WallEditorCanvas.tsx:48`, `backend/app/services/mask_service.py:40`, `backend/app/services/reconstruction_service.py:54`, `backend/app/processing/pipeline.py:365`, `backend/app/processing/pipeline.py:880`, `backend/app/processing/nav_graph.py:20`. В стандартах проекта координаты после vectorization должны быть нормализованы в диапазон `[0, 1]`, а `processing/` должна содержать чистые функции без side effects; в текущем коде часть логики находится в сервисах и API-роутерах.

Для проверки смещения особенно важны: единый источник истины для crop/rotation, использование сохранённого `crop_rect`/rotation в `VectorizationResult`, преобразование координат между изображением, canvas и normalized space, а также повторное применение тех же параметров в preview mask, saved mask, reconstruction и nav graph. В коде уже есть явные места, где эти параметры хранятся и передаются, но также есть несколько путей с placeholder-логикой и дублированием моделей.

## Closest Analog Feature

- Files: `backend/app/api/reconstruction.py:45-330`, `backend/app/services/mask_service.py:24-219`, `backend/app/services/reconstruction_service.py:42-277`, `backend/app/services/nav_service.py:26-146`, `frontend/src/hooks/useWizard.ts:43-158`.
- Data flow: frontend wizard sends file/crop/rotation to `/reconstruction/initial-masks` or `/reconstruction/mask-preview`; mask service loads image and applies crop/rotation/threshold/text removal; reconstruction service reads saved mask and builds `VectorizationResult`, normalizes coordinates, stores JSON and exports mesh; nav service reads mask, builds graph, writes JSON; frontend uses returned IDs/URLs for later steps.

## Integration Points
- Database: `uploaded_files`, `reconstructions`, `rooms`, plus `floors`/`buildings` if floor/building assembly touches the same plans.
- File storage: `backend/app/core/config.py:28` sets `UPLOAD_DIR`; services save under `uploads/plans`, `uploads/masks`, `uploads/models`, `uploads/processed`, `uploads/environment`; generated files use UUID-based names and derived artifacts such as `{file_id}.png`, `{file_id}_text.json`, `reconstruction_{id}.obj/.glb`, `{mask_id}_nav.json`.
- API: frontend `apiService.ts` calls `/upload/plan-photo/`, `/upload/user-mask/`, `/reconstruction/initial-masks`, `/reconstruction/mask-preview`, `/reconstruction/reconstructions`, `/reconstruction/nav-graph`, `/reconstruction/route`.
- Pipeline: current order in standards is upload → preprocessing → wall vectorization → text removal → FloorPlan model → 3D build → editor → pathfinding; current implementation splits the logic across `mask_service`, `reconstruction_service`, and `nav_service`.
