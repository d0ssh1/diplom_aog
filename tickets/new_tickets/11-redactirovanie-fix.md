Анализ корневых причин
Баг 1: Смещение наложения плана
Корень проблемы: В WallEditorCanvas.tsx план-оверлей (<img>) позиционируется через bgDims — координаты фона (маски) на canvas. Однако маска на backend уже является кропнутой и повёрнутой, а на фронтенде план проходит вторичную обработку rotation+crop. При этом:

Маска загружается из URL /api/v1/uploads/masks/{id}.png — уже кропнутая и повёрнутая бэкендом.
Оверлей плана (planUrl) — оригинальное фото до кропа/поворота. Фронтенд применяет rotation и crop вручную (строки 82–113).
Координаты bgDims правильно описывают позицию маски на canvas, но оверлей рисуется как <img> поверх canvas с теми же координатами — это корректно только если размеры оверлея совпадают с маской.
Фактическая ошибка: При отсутствии planCropRect или при некорректном crop (если backend и frontend применяют crop по-разному), размер оверлея не совпадает с размером маски, и overlay выходит за границы маски.

Баг 2: Некорректное восстановление аннотаций
Корень проблемы в двух местах:

2a. Координаты комнат
В EditPlanPage.tsx (строки 46-61) комнаты конвертируются из polygon (нормализованные [0,1]) в RoomAnnotation (x, y, width, height). Координаты x, y, width, height — нормализованные относительно изображения маски.

В restoreAnnotations() (строки 156-228) они преобразуются в canvas-координаты:

ts
const x = leftOffset + room.x * scaledW;
const y = topOffset + room.y * scaledH;
Это корректно. Но проблема в том, что restoreAnnotations() вызывается при загрузке маски и зависит от initialRooms через useCallback. Если маска загрузилась до того, как данные с сервера пришли, то initialRooms будет undefined и ничего не восстановится.

Кроме того, проверка if (canvas.getObjects().length > 0) return; на строке 159 предотвращает повторное восстановление, если маска перезагрузилась.

2b. Двери сохраняются с вырожденными координатами
В EditPlanPage.tsx (строки 63-70) двери конвертируются так:

ts
x1: d.position?.x ?? 0,
y1: d.position?.y ?? 0,
x2: d.position?.x ?? 0,  // x2 == x1!
y2: d.position?.y ?? 0,  // y2 == y1!
Двери — это точки (circles), не линии, так что x1==x2, y1==y2 корректно для точечной модели. Но в restoreAnnotations() дверь позиционируется:

ts
const cx = leftOffset + ((door.x1 + door.x2) / 2) * scaledW;
const cy = topOffset + ((door.y1 + door.y2) / 2) * scaledH;
Это должно работать. Проблема скорее в том, что API возвращает 0 дверей — значит, при сохранении двери не были записаны в vectorization_data.

2c. saveVectors не вызывается при начальном сохранении через Wizard
В WizardPage.tsx (строки 30-37), saveVectors вызывает saveMaskAndAnnotations(), который сохраняет аннотации в wizard state, а потом buildNavGraph(). Но вектризация-данные (vectorization_data) обновляются только через reconstructionApi.updateVectorizationData(), а в wizard flow это не вызывается — вместо этого rooms и doors передаются напрямую в buildNavGraph().

Вывод: При создании плана через wizard, аннотации rooms/doors не сохраняются в vectorization_data. Они используются только для построения графа. Поэтому при редактировании данных нет.

Предложенные изменения
Компонент 1: Сохранение аннотаций в vectorization_data при создании
[MODIFY] 
useWizard.ts
В методе saveMaskAndAnnotations() — после сохранения маски, дополнительно обновлять vectorization_data на бэкенде, включая rooms и doors. Это гарантирует, что при следующем редактировании данные будут доступны.

Компонент 2: Исправление восстановления аннотаций
[MODIFY] 
WallEditorCanvas.tsx
Убрать ранний return if (canvas.getObjects().length > 0) return; — он предотвращает повторное восстановление при обновлении initialRooms/initialDoors.
Очищать старые аннотации перед восстановлением, чтобы при обновлении props не дублировались объекты.
Корректная зависимость restoreAnnotations от initialRooms и initialDoors + bgDims — нужно вызывать восстановление когда оба готовы (и данные, и размеры фона).
Компонент 3: Исправление смещения наложения
[MODIFY] 
WallEditorCanvas.tsx
Проблема в том, что бэкенд уже применяет rotation и crop к маске, но у фронтенда нет информации о точных пиксельных размерах результирующей маски до тех пор, пока маска не загрузится в canvas. Когда оверлей подготавливается из оригинала с rotation+crop, его proportions могут не совпадать с маской из-за разницы в интерполяции.

Решение: Привязать размеры оверлея не к bgDims, а рассчитать их из соотношения сторон реальной маски, обеспечивая pixel-perfect совпадение.

Конкретно:

Когда displayPlanUrl генерируется (строки 82-113), он всегда должен быть точно того же aspect ratio, что и маска. Сейчас это не гарантировано, потому что crop координаты — нормализованные, и при пересчёте в пиксели с Math.round могут быть расхождения.
Фикс: После загрузки маски (bgDims обновлены), при генерации оверлея из оригинала — подгонять финальный размер кропнутого оверлея под aspect ratio маски.
